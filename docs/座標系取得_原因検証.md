# 座標系取得失敗の原因検証

## 状況

`merge_gaiku_geoparquet.py` 実行時に、多くの H_*.parquet が「skip (no 座標系 1-19)」となる。一方、`--dump-ogrinfo` で同じファイルの ogrinfo 出力を確認すると、**座標系 (String) = 12** などが明らかに存在する。

## 検証手順

### 1. 検証スクリプトの実行

```bash
cd /home/ubuntu/work/cursor/maplibre/MapLibre-HandsOn-Beginner/05_ポリゴン表示/gdal-full
source env.sh
python3 scripts/verify_zukei_debug.py
```

または、特定ファイルを指定:

```bash
python3 scripts/verify_zukei_debug.py "inputfile/.../街区基準点等データ/H_01101.parquet"
```

### 2. 確認すべき項目（stderr に出力）

| 項目 | 期待値 | 確認内容 |
|------|--------|----------|
| ogrinfo returncode | 0 | 正常終了しているか |
| stdout/stderr length | 0 以外 | 出力がキャプチャされているか |
| 1) 座標系 直接 | NO MATCH（文字化けのため） | 日本語列名が使えない理由 |
| 2) Field 10/11/12 | NO MATCH | ogrinfo が "Field N:" 形式を出していない |
| 3) OGRFeature ブロック | MATCH | 最初のフィーチャが取得できているか |
| block 先頭 500 文字 | 属性行が含まれる | ブロック内容が正しいか |
| attr_vals 件数 | 23 前後 | 属性行が正しくパースされているか |
| idx=11 の val | "12" | 座標系の値が取れているか |

### 3. 想定される原因候補

1. **subprocess の出力キャプチャ**
   - ogrinfo が INFO 等を stderr に、フィーチャを stdout に出す場合、結合順序や欠損の可能性
   - `encoding="utf-8", errors="replace"` による文字化け・置換で正規表現がマッチしない

2. **OGRFeature ブロックのマッチ失敗**
   - 正規表現 `OGRFeature\([^)]+\):\d+\s*\n(.*?)(?=OGRFeature\(|\Z)` が、実際の出力形式と合っていない
   - 例: レイヤ名に括弧が含まれる、改行コードの違い

3. **属性行のパース失敗**
   - `\)\s*=\s*(.*)$` が一部の行にマッチしない（値に改行が含まれる、行末の \r 等）
   - その結果 `attr_vals` の件数が少なく、idx 11 が座標系を指していない

4. **ファイルパス・cwd**
   - `cwd=PROJECT_ROOT` のとき、相対パス指定で ogrinfo が別ファイルを読んでいる可能性
   - 日本語を含むパスで subprocess に渡す際のエンコーディング

## 次のアクション

検証スクリプトの出力を確認し、上記のどの段階で失敗しているかを特定する。原因が分かったら、`merge_gaiku_geoparquet.py` の `get_zukei` を最小限の修正で対応する。
